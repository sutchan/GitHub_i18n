# 文档格式规范

## 1. 目的

本文档定义了GitHub中文翻译插件(GitHub_i18n)项目文档的格式规范，确保所有文档具有一致性和可读性，便于团队协作和文档维护。

## 2. 文档分类

### 2.1 基础文档
- **README**：项目文档首页，提供整体导航和项目概述
- **文档格式规范**：定义文档编写标准和格式要求
- **文档索引**：提供文档导航和分类
- **文档审计与清理报告**：记录文档审计和清理工作
- **文档版本管理**：定义文档版本控制规则
- **文档检索系统**：提供文档搜索和检索指南
- **TODO任务列表**：记录项目待办事项和计划

### 2.2 用户指南
- **使用说明**：提供插件的安装、配置和基本使用方法
- **FAQ**：常见问题解答
- **配置选项**：详细说明插件的配置参数和选项

### 2.3 开发文档
- **开发指南**：开发环境搭建、流程和注意事项
- **构建指南**：项目构建流程和命令
- **API文档**：描述API接口、参数和返回值
- **页面模式支持与性能优化**：技术实现细节文档

### 2.4 架构设计
- **系统架构文档**：描述系统的整体架构和组件关系
- **工作流文档**：记录开发和发布工作流程

### 2.5 测试报告
- **功能测试报告**：记录功能测试的结果和问题
- **性能测试报告**：记录性能测试的结果和优化建议
- **工作流审查报告**：工作流程的审查结果

### 2.6 更新日志
- **版本更新历史**：记录各版本的更新内容和变更说明

### 2.7 归档目录
- **归档策略**：定义文档归档的规则和流程
- **历史版本**：存储历史版本的文档

## 3. 文档格式要求

### 3.1 标题层级
- **H1 (#)**：文档标题，每个文档必须且只能有一个H1
  - 示例：`# 文档格式规范`
- **H2 (##)**：主要章节标题，用于划分文档的主要部分
  - 示例：`## 1. 目的`
- **H3 (###)**：子章节标题，用于进一步细分内容
  - 示例：`### 2.1 基础文档`
- **H4 (####)**：子子章节标题，用于更详细的内容划分
  - 示例：`#### 3.1.1 标题格式示例`
- **H5 (#####)**：很少使用，除非需要更细致的层级
- **H6 (######)**：基本不使用

### 3.2 文本格式
- **加粗 (***)**：用于强调重要内容、标题中的关键字
  - 示例：`***重要：*** 请确保按照此规范执行`
- **斜体 (*)**：用于辅助说明或特殊术语首次出现
  - 示例：`*DOM* 即文档对象模型`
- **删除线 (~~)**：用于标记已废弃的内容
  - 示例：`~~此功能已废弃~~`
- **行内代码 (`)**：用于表示代码、变量名、命令、路径等
  - 示例：`npm install` 命令用于安装依赖

### 3.3 代码块与引用
- **代码块 (```language)**：用于展示代码片段，必须指定编程语言
  ```javascript
  function translateElement(element) {
    if (element && element.textContent) {
      element.textContent = getTranslation(element.textContent);
    }
  }
  ```
- **命令行代码块 (```bash)**：用于展示终端命令
  ```bash
  git clone https://github.com/sutchan/GitHub_i18n.git
  cd GitHub_i18n
  ```
- **引用 (> )**：用于引用外部内容或重要说明性文字
  > **注意：** 本功能需要浏览器支持ES6+特性

### 3.4 列表与表格
- **无序列表 (-)**：用于没有特定顺序的列表项
  - 特性一
  - 特性二
  - 特性三
- **有序列表 (1.)**：用于有顺序要求的列表项
  1. 首先执行安装
  2. 然后进行配置
  3. 最后启动服务
- **任务列表 (- [ ])**：用于跟踪任务完成情况
  - [x] 完成文档格式规范
  - [ ] 更新API文档
  - [ ] 编写用户指南
- **表格 (| 表头 | 表头 |)**：用于展示结构化数据
  | 参数名 | 类型 | 必填 | 说明 |
  |-------|------|------|------|
  | timeout | number | 否 | 超时时间(ms) |
  | retry | boolean | 否 | 是否自动重试 |

### 3.5 图片与媒体
- **图片格式**：优先使用PNG或SVG格式
- **图片大小**：确保图片清晰可读，大小适中（建议不超过2MB）
- **图片命名**：使用描述性文件名，采用小写字母和下划线组合
- **图片引用**：使用Markdown的图片语法，并添加Alt文本
  ```markdown
  ![示例图片](images/example.png)
  ```
- **图片路径**：统一存放在文档根目录的images文件夹中

## 4. 文档命名规范

### 4.1 文件名格式
- 使用中文描述性文件名，避免使用缩写
- 文件名应清晰表达文档内容和用途
- 文件名中使用空格分隔单词
- 示例：`项目架构设计.md`、`API接口文档.md`

### 4.2 路径规范
- 按照文档分类将文档存放在对应子目录中
- 避免过深的目录层级，建议不超过3级
- 保持目录结构清晰，便于导航和检索
- 子目录命名使用中文，保持一致性

## 5. 文档版本信息

### 5.1 版本号格式
- 遵循语义化版本号规范：MAJOR.MINOR.PATCH
  - MAJOR：不兼容的重大变更
  - MINOR：向下兼容的功能性新增
  - PATCH：向下兼容的问题修复
- 初始版本从1.0.0开始
- 文档格式更新或结构调整时，更新MINOR版本号
- 修复错误或少量内容更新时，更新PATCH版本号

### 5.2 版本信息标注
- 每个文档必须包含版本信息
- 版本信息应放在文档末尾
- 包含以下信息：版本号、最后更新日期、作者、状态
- 格式示例：
  ```markdown
  ## 版本信息

  版本：1.0.0 - 最后更新：2024-06-02
  作者：Sut
  状态：已发布
  ```

## 6. 文档内容要求

### 6.1 内容结构
- 文档应包含清晰的标题和章节结构
- 内容应逻辑连贯，层次分明
- 重点内容应突出显示
- 每个章节应包含简短的说明性文字
- 长文档应在开始处提供目录导航

### 6.2 语言规范
- 使用简洁明了的语言
- 术语使用应保持一致，避免中英文混用（除专有名词外）
- 避免使用模糊不清的表述
- 针对技术文档，应使用准确的技术术语
- 使用第二人称称呼读者（"您"）
- 指令性内容使用祈使句，明确指导操作

### 6.3 内容质量
- 内容应准确无误，与实际代码和功能保持一致
- 内容应完整无缺，涵盖所有必要信息
- 内容应及时更新，反映最新的功能和变化
- 应提供足够的上下文和背景信息
- 对于复杂概念，应提供示例或图示说明

## 7. 文档审核与发布流程

### 7.1 审核流程
- 文档创建或修改后，应由相关人员审核
- 审核内容包括：格式规范、内容准确性、完整性、语言表达等
- 审核通过后，文档可正式发布
- 重大文档变更应由项目负责人审核

### 7.2 发布流程
- 文档发布后，应在文档索引中更新文档状态和版本信息
- 重要文档的更新应通知相关团队成员
- 发布的文档应保持稳定，避免频繁变更
- 文档状态变更应记录在版本信息中

## 8. 文档维护指南

### 8.1 更新频率
- 核心文档（如API文档、使用指南）应与代码同步更新
- 其他文档应定期审查和更新，建议每季度至少一次
- 项目有重大变更时，应立即更新相关文档

### 8.2 文档状态管理
- **草稿**：文档正在编写中，尚未完成
- **审核中**：文档已完成初稿，正在进行审核
- **已发布**：文档已通过审核，可以正式使用
- **已归档**：文档已不再使用，但出于历史原因保留
- **已废弃**：文档已完全过时，将在适当时候删除

### 8.3 归档与清理
- 过时但可能有参考价值的文档应归档到归档目录
- 无实际价值的文档应定期清理
- 文档归档和清理应记录在文档审计报告中

## 9. 附录

### 9.1 模板示例
- **基础文档模板**：
  ```markdown
  # 文档标题

  ## 1. 目的
  简要说明文档的目的和作用。

  ## 2. 内容概述
  概述文档包含的主要内容。

  ## 3. [具体内容章节]
  根据文档类型添加相应章节。

  ## 版本信息

  版本：1.0.0 - 最后更新：YYYY-MM-DD
  作者：[姓名]
  状态：[草稿/审核中/已发布/已归档/已废弃]
  ```
